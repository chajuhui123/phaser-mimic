# Technical Direction

이 프로젝트는 React, Next.js, TypeScript, Three.js 기반의 1인칭 시점 3D 멀티플레이어 웹 게임이다.

추후 WebView로 감싸 iOS/Android 모바일 앱 게임으로 확장할 예정이다.

## Tech Stack

- Next.js
- React
- TypeScript
- Three.js (`@react-three/fiber`, `@react-three/drei`)

## Responsibility Split

### Next.js

Next.js는 게임 외부의 앱 화면을 담당한다.

- 랜딩/홈
- 로그인
- 로비
- 방 목록
- 방 생성/입장
- 티켓 상태
- 결과 화면
- 설정 화면

### Three.js (3D 렌더링)

Three.js(`@react-three/fiber`)는 실제 게임 플레이 화면을 담당한다.

- 3D 맵/조명 렌더링
- 카메라 처리 (마우스로 시점을 돌리지 않고, 항상 고정된 정면을 바라보며 캐릭터를 따라 위치만 이동함 — 자신 캐릭터의 뒷모습이 화면 아래쪽에 보임)
- 캐릭터 이동
- NPC 이동 및 미션성 랜덤 행동
- 플레이어 입력 (키보드 이동)
- 술래 공격
- 충돌 판정
- 술래 근접 경고 이펙트
- 미션 연출
- 강제 변신 상태 표현
- 게임 HUD (프로필, 타이머, 기여도)
- 타이머 표시

### Realtime Server

Realtime Server는 실시간 게임 상태를 담당한다.

- 방 상태
- 플레이어 입장/퇴장
- 역할 배정
- 플레이어 위치 동기화
- NPC 상태 동기화
- 미션 발급
- 공격 판정
- 체력 변경
- 탈락 처리
- 승패 판정

### Database

Database는 영속 데이터를 저장한다.

- 유저 정보
- 티켓 수량
- 전적
- 매치 로그
- 게임 결과

## Recommended Initial Architecture

초기에는 서버를 바로 완성하지 않고, 로컬 목업 상태로 3D 프로토타입을 먼저 만든다.

1. Next.js 프로젝트 생성
2. Three.js(`@react-three/fiber`) 3D 게임 화면 연결
3. 로컬 mock room state 생성
4. 플레이어/NPC 렌더링 (초기에는 primitive 3D 모형 사용, 실제 모델은 이후 교체)
5. 술래/도망자 역할 시뮬레이션
6. 공격/피격/오공격 처리
7. 미션 시스템 구현
8. 미션 실패 시 강제 변신 구현
9. 타이머와 승패 판정 구현
10. 이후 실시간 서버로 상태 관리 이전

## Development Principle

첫 목표는 완성도 높은 서비스가 아니라 재미 검증이다.

따라서 초기 구현에서는 아래를 우선한다.

- 빠른 프로토타입
- 단순한 룰
- 명확한 승패
- 플레이 가능한 한 판
- 모바일 확장을 고려한 입력 구조
- 서버 구현 전에도 로컬에서 게임 루프 검증 가능

## Future Considerations (지금 만들지 않지만 구조적으로 대비할 것)

아래 항목은 지금 단계에서 직접 구현하지 않는다. 다만 이 프로젝트는 최종적으로 앱 출시(WebView 패키징)를 목표로 하므로, 이 항목들이 나중에 자연스럽게 얹힐 수 있도록 지금 구조를 짤 때 하드코딩으로 막아두지 않는다.

- **게임 스토리텔링**: 지금은 룰만 있고 서사가 없다. 미션/역할/HUD에 노출되는 텍스트를 코드에 하드코딩하지 않고 데이터(별도 파일/상수)로 분리해, 이후 세계관/카피가 들어와도 로직을 건드리지 않고 교체할 수 있게 한다.
- **구조 설계(콘텐츠 확장)**: 맵·미션·역할 종류가 지금은 하나뿐이어도 여러 개로 늘어날 것을 가정한 타입/데이터 구조를 쓴다 (예: 맵을 단일 상수가 아니라 목록 중 하나로 다루기).
- **레벨 디자인**: 지금 맵은 임시 mock이다. 이후 여러 맵/난이도가 추가될 수 있으므로 맵 정의(지형, NPC 배치, 스폰 지점)를 씬 코드와 분리된 데이터로 둔다.
- **재화 시스템**: 지금은 티켓(참여 자원) 하나만 존재하지만, 이후 결제형 재화나 보상 재화가 추가될 수 있다. 티켓 로직을 독립된 모듈로 분리해 재화 종류가 늘어나도 각각 별도로 확장 가능하게 한다.
- **멀티 서버 세팅**: 지금은 로컬 mock 상태다. 위치/역할/체력 등 게임 상태를 다루는 코드가 "지금은 로컬 변수, 나중엔 서버 동기화 값"이라는 전제를 깨지 않도록, 게임 엔진 코드가 상태를 직접 소유하지 않고 별도 상태 계층을 통해서만 접근하게 한다.
- **배포/런칭**: 지금은 `next.config.mjs`의 `output: 'export'`로 정적 빌드하지만, 이후 WebView 앱 패키징을 고려해 `window`/절대 URL 등 브라우저 전용 API 의존을 최소화하고, 클라이언트 전용 코드 경계(`ssr: false`)를 명확히 유지한다.

## Mobile Consideration

추후 WebView 앱으로 확장할 예정이므로 초기부터 모바일 환경을 고려한다.

- 세로 또는 가로 화면 방향을 명확히 정한다.
- 터치 입력을 우선 고려한다.
- 버튼 크기는 모바일 터치에 적합해야 한다.
- HUD는 작은 화면에서도 읽기 쉬워야 한다.
- 게임 화면은 다양한 해상도에 대응해야 한다.
