# MVP 6단계 작업 Command

당신은 1인칭 시점의 3D 멀티플레이 심리 추리 액션 게임을 개발하는 역할입니다.

코드를 수정하기 전에 먼저 아래 문서를 읽으세요.

- docs/game-concept.md
- docs/mvp-scope.md
- docs/technical-direction.md
- docs/mvp-roadmap.md

위 문서들을 프로젝트의 기준 문서로 사용하세요.

## 프로젝트 배경

MVP 5단계까지 이동/시야 회전, 술래 공격/체력/탈락, hider 근접 경고/기여도, 미션과 강제 변신을 구현했습니다. 지금까지는 게임이 시작/종료 없이 계속 "플레이 중" 상태로 열려 있었습니다. 이번 MVP 6단계는 한 판의 시작(대기·카운트다운·티켓 소모)과 종료(승패 판정·결과 화면·관전 모드)를 추가해 "한 판"이라는 개념을 완성합니다.

게임 자체의 콘셉트/역할/규칙은 위에서 읽은 `docs/game-concept.md`, `docs/mvp-scope.md`를 기준으로 삼으세요 (여기서 다시 요약하지 않습니다).

기존 역할 전환 디버그 기능(`src/game3d/debug/*`)과 역할별 HUD 분리 패턴(`CombatHud`, `MissionHud`)을 재사용하세요. 게임 진행 단계(대기/카운트다운/플레이/종료)는 새로운 최상위 상태로 두고, 기존 이동/공격/미션 로직은 "플레이 중" 단계에서만 동작하도록 감싸야 합니다.

## 공통 규칙

작업 중 아래 문서의 규칙을 따르세요.

- `CLAUDE.md` — 클라이언트 전용 렌더링, 게임 엔티티/mock 데이터 분리 구조
- `docs/mvp-scope.md`의 `Excluded` 목록
- `docs/technical-direction.md`의 "Future Considerations"

추가로 아래 두 가지는 이번 커맨드에 고유한 규칙이니 항상 지키세요.

- 전체 게임을 한 번에 구현하지 않고, 현재 요청받은 MVP 단계의 범위만 구현합니다.
- 구현 후 변경 파일, 실행 방법, 확인 방법을 정리합니다.

## MVP 6단계 목표

이번 작업에서는 MVP 6단계만 구현합니다. (`docs/mvp-roadmap.md` 기준 "한 판의 시작/종료/승패 흐름이 완성된다" 단계입니다.)

MVP 6단계 목표는 다음과 같습니다.

- 게임 시작 전 대기 상태와 시작 카운트다운이 있습니다.
- 게임 시작 시 티켓을 5개 소모하며, 티켓이 부족하면 시작할 수 없습니다 (티켓 최대 보유량 50, 시간 경과에 따른 자동 충전 로직 포함 — 결제는 제외).
- 제한 시간 5분 타이머가 동작합니다.
- 상단 HUD에 술래/도망자 프로필과 남은 시간이 표시됩니다.
- 아래 승패 조건이 판정됩니다.
  - 모든 hider가 탈락하면 seeker 팀 승리
  - 제한 시간 종료 시 hider가 1명 이상 생존하면 hider 팀 승리
  - seeker 체력이 0이 되면 hider 팀 승리
- 게임 종료 후 결과 화면(승리 팀, 주요 결과)을 표시합니다.
- 탈락한 유저는 관전 모드로 전환됩니다.
- 실제 방/로비 서버, 랭킹, 보상 지급, 티켓 구매(결제), 장기 전적 저장은 아직 구현하지 않습니다.

## 작업 방식

### Step A: 문서 읽고 요약하기

먼저 아래 문서를 읽으세요.

- docs/game-concept.md
- docs/mvp-scope.md
- docs/technical-direction.md
- docs/mvp-roadmap.md

그다음 아래 항목을 요약하세요.

- 게임의 핵심 콘셉트
- MVP 5단계에서 이미 구현된 내용 (현재 코드 기준)
- MVP 6단계에 포함되는 기능
- MVP 6단계에서 제외되는 기능
- Next.js와 Three.js의 역할 분리
- 구현 시 주의해야 할 점 (특히 현재 mock 구성이 hider 1명뿐이라는 점이 관전 대상 선택 기능에 미치는 영향)

이 단계에서는 파일을 수정하지 마세요.

### Step B: 작업 계획 제안하기

문서 요약 후 MVP 6단계 구현 계획을 제안하세요.

아래 항목을 포함하세요.

- 작업 범위
- 제외 범위
- 예상 파일 구조(신규/수정 파일 — 매치 상태 머신, 티켓 상태, 타이머, 결과 화면, 관전 모드)
- 구현 순서
- 리스크 또는 결정이 필요한 부분 (예: hider가 1명뿐인 mock에서 관전 대상 선택 UI를 어떻게 단순화할지, 티켓 자동 충전 주기를 테스트 가능하게 어떻게 잡을지, 대기/카운트다운 중 기존 디버그 역할 전환 기능을 어떻게 다룰지)

이 단계에서도 파일을 수정하지 마세요.

### Step C: 승인 기다리기

파일을 수정하기 전에 반드시 명시적인 승인을 기다리세요.

아래와 같은 승인 표현이 있을 때만 구현을 시작하세요.

- "진행해"
- "구현해"
- "좋아 진행"
- "Proceed"
- "Implement it"
- "Go ahead"

### Step D: MVP 6단계 구현하기

승인을 받은 뒤 MVP 6단계만 구현하세요.

구현 방향은 다음과 같습니다.

- 기존 MVP 5단계 구조(`src/game3d/*`)를 최대한 재사용하고 확장합니다.
- 매치 진행 단계(대기/카운트다운/플레이/종료)를 최상위 상태로 두고, 기존 이동/공격/미션 로직은 "플레이 중" 단계에서만 동작하게 감쌉니다.
- 티켓/타이머/승패/결과 상태는 이후 실시간 서버 상태로 교체할 수 있도록 구조화합니다.
- 결과 화면, 관전 모드 등 새 UI도 하드코딩 텍스트 대신 필요한 부분만 데이터로 분리합니다.
- 실제 방/로비 서버, 랭킹, 보상 지급, 결제, 장기 전적 저장은 이번 단계에서 구현하지 않습니다.

### Step E: 검증하기

구현 후 프로젝트를 검증하세요.

아래 항목을 확인하세요.

- `/game` 페이지가 정상적으로 렌더링되는지
- 대기 상태에서 카운트다운을 거쳐 게임이 시작되는지
- 게임 시작 시 티켓이 5개 소모되고, 부족하면 시작할 수 없는지
- 5분 타이머와 상단 HUD(프로필/남은 시간)가 표시되는지
- 세 가지 승패 조건이 각각 정상적으로 판정되는지
- 게임 종료 후 결과 화면이 표시되는지
- 탈락 시 관전 모드로 전환되는지
- 디버그 역할 전환, 기존 이동/공격/미션 기능이 "플레이 중" 단계에서 여전히 정상 동작하는지
- hot reload 또는 컴포넌트 remount 시 3D 캔버스/렌더러가 중복 생성되지 않는지
- TypeScript 타입 에러가 없는지
- 가능한 경우 lint/build 에러가 없는지 확인하고 수정했는지

### Step F: 결과 보고하기

검증 후 아래 내용을 정리해서 보고하세요.

- 변경한 파일 목록
- 구현한 내용
- 프로젝트 실행 방법
- `/game` 페이지 확인 방법 (대기/카운트다운/승패/결과/관전 확인 방법 포함)
- 의도적으로 아직 구현하지 않은 기능
- 추천하는 다음 단계(After MVP) 작업

## 응답 방식

응답은 간결하지만 명확하게 작성하세요.

계획을 제안하는 단계에서는 코드를 작성하지 마세요.

구현 단계에서는 범위를 좁게 유지하세요.

결과 보고에서는 실제로 확인할 수 있는 내용 중심으로 정리하세요.
