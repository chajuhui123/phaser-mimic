# 웹뷰 연동을 위한 화면 최소 세팅 Command

당신은 1인칭 시점의 3D 멀티플레이 심리 추리 액션 게임을 개발하는 역할입니다.

코드를 수정하기 전에 먼저 아래 문서를 읽으세요.

- docs/game-concept.md
- docs/mvp-scope.md
- docs/technical-direction.md
- docs/mvp-roadmap.md

위 문서들을 프로젝트의 기준 문서로 사용하세요.

## 프로젝트 배경

MVP 1~6단계를 거쳐 `/game` 페이지에서 한 판(대기 → 카운트다운 → 플레이 → 종료/결과)이 동작하는 3D 게임 화면은 완성되어 있습니다. 하지만 그 게임 화면으로 들어가기 전의 "앱 외부 화면"은 아직 없습니다. `docs/technical-direction.md`의 Responsibility Split에 따르면 Next.js는 랜딩/홈, 로그인, 로비 같은 게임 외부 화면을 담당하도록 역할이 나뉘어 있는데, 지금은 `/` 페이지가 제목과 링크 하나뿐인 임시 페이지입니다.

또한 이 프로젝트는 최종적으로 WebView로 감싸 iOS/Android 앱으로 출시할 계획이며(`docs/technical-direction.md`의 Mobile Consideration 참고), 화면 방향은 세로로 고정하기로 정했습니다. 지금은 브라우저에서만 확인하지만, 나중에 실제 WebView 앱으로 패키징했을 때 화면 방향/레이아웃이 깨지지 않도록 지금부터 최소한의 앱 셸(스플래시 → 로비 → 게임 진입 흐름, 세로 프레임 레이아웃)을 갖춰두는 것이 이번 작업의 목적입니다.

게임 플레이(3D 씬 안의 이동/공격/미션/승패 등)는 이번 작업 범위가 아닙니다. `src/game3d/*`의 기존 게임플레이 로직은 건드리지 마세요.

## 공통 규칙

작업 중 아래 문서의 규칙을 따르세요.

- `CLAUDE.md` — 클라이언트 전용 렌더링, React state/props 우선 사용(불필요한 이벤트 브릿지 금지)
- `docs/mvp-scope.md`의 `Excluded` 목록
- `docs/technical-direction.md`의 Responsibility Split과 "Future Considerations", Mobile Consideration

추가로 아래는 이번 커맨드에 고유한 규칙이니 항상 지키세요.

- 이번 작업은 "앱 셸(게임 외부 화면 + 레이아웃)"만 다룹니다. 3D 게임플레이 로직, 실시간 서버, 로그인, 실제 방 목록/방 생성, 결제는 이번 범위가 아닙니다.
- 실제 서버나 계정 시스템이 없는 상태이므로, 화면 전환은 정적 라우팅(버튼 → 다음 페이지 이동) 수준의 목업으로 구현합니다.
- 세로 모드 강제는 사용자에게 "회전해달라"는 경고 문구로 막는 방식이 아니라, 화면을 그대로 세로로 유지하도록 처리합니다 (가로로 기기를 돌려도 콘텐츠를 회전시켜 세로 형태를 유지).
- 나중에 실제 서버 상태/로그인/방 목록으로 교체 가능하도록, 화면 전환에 쓰이는 값(예: 최대 너비 상수)은 하드코딩 대신 상수/모듈로 분리합니다.

## 이번 커맨드 목표

이번 작업에서는 아래 "앱 셸 최소 스펙"만 구현합니다.

- 앱 진입 시 스플래시 화면이 보이고, "시작하기" 버튼을 눌러야 페이드 아웃/페이드 인 트랜지션으로 로비 화면으로 전환됩니다. (자동으로 사라지는 타이머 방식이 아닙니다.)
- 로비 화면에서 "게임 시작"을 누르면 기존 `/game` 페이지(3D 게임 화면)로 이동합니다.
- 모바일 세로 화면 전용 게임이므로, 데스크톱 브라우저처럼 화면이 넓은 환경에서도 세로 모바일 폭을 넘지 않는 프레임 안에서만 렌더링됩니다.
- 실제 기기를 가로로 돌려도 경고 문구로 막지 않고, 콘텐츠를 회전시켜 화면을 세로 형태로 그대로 유지합니다 (추후 WebView에서 네이티브 방향 고정으로 대체할 수 있는 구조로 둡니다).

아래는 이번 범위에서 제외합니다.

- 로그인/계정 시스템
- 실제 방 목록/방 생성/입장 서버 연동
- 티켓 구매(결제)
- 스플래시/로비의 브랜딩 디자인 다듬기(로고, 세부 비주얼 등) — 버튼 클릭 시 페이드 전환 외의 추가 연출은 제외합니다.
- 3D 게임플레이 로직 변경

## 작업 방식

### Step A: 문서 읽고 요약하기

먼저 아래 문서를 읽으세요.

- docs/game-concept.md
- docs/mvp-scope.md
- docs/technical-direction.md
- docs/mvp-roadmap.md

그다음 아래 항목을 요약하세요.

- 게임의 핵심 콘셉트
- 현재까지 구현된 내용(MVP 1~6단계, `/game` 페이지 기준)과 지금 비어 있는 부분(앱 외부 화면)
- Next.js와 Three.js의 역할 분리, 그중 이번 작업이 담당하는 부분
- Mobile Consideration에서 이번 작업과 관련된 항목(화면 방향, 다양한 해상도 대응)
- 구현 시 주의해야 할 점 (특히 실제 WebView 패키징 전까지는 브라우저에서만 검증 가능하다는 점)

이 단계에서는 파일을 수정하지 마세요.

### Step B: 작업 계획 제안하기

문서 요약 후 구현 계획을 제안하세요.

아래 항목을 포함하세요.

- 작업 범위와 제외 범위
- 예상 파일 구조(신규/수정 파일 — 스플래시/로비 화면, 공통 앱 셸 레이아웃, 전역 스타일)
- 화면 전환 방식(라우팅 vs 내부 상태)과 그 이유
- 세로 프레임 최대 너비, 가로 회전 시 처리 방식에 대한 구체적인 방법
- 리스크 또는 결정이 필요한 부분

이 단계에서도 파일을 수정하지 마세요.

### Step C: 승인 기다리기

파일을 수정하기 전에 반드시 명시적인 승인을 기다리세요.

아래와 같은 승인 표현이 있을 때만 구현을 시작하세요.

- "진행해"
- "구현해"
- "좋아 진행"
- "Proceed"
- "Implement it"
- "Go ahead"

### Step D: 구현하기

승인을 받은 뒤 앱 셸 최소 스펙만 구현하세요.

구현 방향은 다음과 같습니다.

- 게임 외부 화면(스플래시, 로비)은 `src/game3d/*`와 분리된 별도 위치에 둡니다.
- 스플래시("/") → 로비("/lobby") → 게임("/game") 순서로 라우팅합니다. 스플래시 → 로비, 로비 → 게임 모두 버튼 클릭으로 전환하며, 스플래시 → 로비 전환에는 페이드 아웃/페이드 인 트랜지션을 적용합니다.
- 모든 페이지를 감싸는 공통 레이아웃에서 세로 프레임 최대 너비를 상수로 정의하고, 데스크톱 등 넓은 화면에서도 그 폭을 넘지 않게 중앙에 렌더링합니다.
- 실제 가로 회전을 감지했을 때는 경고 문구 대신 콘텐츠 자체를 회전시켜 세로 형태를 유지하는 방식으로 처리합니다.
- 기존 `/game` 페이지와 `src/game3d/*` 내부 게임플레이 로직은 변경하지 않습니다(레이아웃에 맞게 감싸는 정도만 허용).

### Step E: 검증하기

구현 후 프로젝트를 검증하세요.

아래 항목을 확인하세요.

- `/`에서 "시작하기" 클릭 시 페이드 트랜지션으로 `/lobby`로 전환되는지 → "게임 시작" → `/game` 흐름이 정상적으로 이어지는지
- 브라우저 창을 넓게 늘려도 화면이 세로 프레임 폭을 넘지 않고 중앙에 렌더링되는지
- 브라우저 반응형 모드에서 모바일 화면 + 가로 방향(landscape)으로 전환했을 때 경고 문구 없이 콘텐츠가 세로 형태로 유지되는지, 다시 세로로 돌리면 정상으로 돌아오는지
- `/game` 페이지의 기존 게임플레이(이동/공격/미션/승패/결과)가 이전과 동일하게 동작하는지
- TypeScript 타입 에러가 없는지
- 가능한 경우 lint/build 에러가 없는지 확인하고 수정했는지

### Step F: 결과 보고하기

검증 후 아래 내용을 정리해서 보고하세요.

- 변경한 파일 목록
- 구현한 내용(화면 흐름, 세로 프레임 처리, 가로 회전 처리 방식)
- 프로젝트 실행 방법
- 화면 흐름과 세로 모드 유지를 직접 확인하는 방법
- 의도적으로 아직 구현하지 않은 기능
- 추천하는 다음 단계 작업(예: 실제 로비/방 목록 서버 연동, WebView 패키징 시 네이티브 방향 고정으로 교체)
